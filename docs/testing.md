# 2.6 Тестирование системы

Тестирование системы поддержки пользователей выделено в отдельный этап, так как проект содержит несколько уровней логики: серверные сервисы, REST API, а также интеллектуальный модуль классификации и подбора рекомендаций. Проверки выполнялись на трёх уровнях: модульные тесты бизнес-логики, интеграционные проверки REST API через Postman/Newman и специализированные сценарии для AI/RAG-компонентов.

## 2.6.1 Модульное тестирование (JUnit, Mockito)

Модульные тесты серверной части реализованы на JUnit 5 с использованием Mockito. Такой подход позволяет проверять бизнес-логику сервисов изолированно от базы данных, внешнего LLM-сервиса и сетевой инфраструктуры. В тестах создаются mock-объекты репозиториев, сервисов уведомлений, SLA и AI-компонентов, после чего проверяются как возвращаемые значения, так и побочные эффекты: сохранение сущностей, запись аудита, создание истории статусов и генерация токенов.

Основные группы модульных тестов:

- `AiServiceTest` — проверка классификации обращения, устойчивого извлечения JSON из ответа модели, обработки фигурных скобок внутри строк JSON и локального fallback-поиска эмбеддингов при недоступности `pgvector`.
- `TaskServiceTest` — проверка создания заявки: установка статуса `NEW`, значений по умолчанию для категории и приоритета, вызов SLA-логики, обновление эмбеддинга, запись истории и аудита, отправка уведомления. Отдельно покрыт негативный сценарий с некорректным дедлайном.
- `AuthServiceTest` — проверка успешного входа, выпуска access/refresh-токенов, сохранения hash refresh-токена вместо исходного значения, отклонения неверного пароля и ротации refresh-токена при обновлении пары токенов.

Пример модульного теста с mock-зависимостями для создания обращения:

```java
@Test
void createTaskShouldSetDefaultsAndCallSideEffects() {
    TaskRepository taskRepository = mock(TaskRepository.class);
    UserRepository userRepository = mock(UserRepository.class);
    SlaService slaService = mock(SlaService.class);
    EmbeddingService embeddingService = mock(EmbeddingService.class);
    NotificationService notificationService = mock(NotificationService.class);

    TaskService service = new TaskService(
            taskRepository,
            userRepository,
            mock(AuditLogRepository.class),
            slaService,
            mock(AiService.class),
            embeddingService,
            mock(TicketCommentRepository.class),
            mock(TicketStatusHistoryRepository.class),
            mock(AiRecommendationRepository.class),
            notificationService,
            new ObjectMapper()
    );

    when(taskRepository.save(any(TaskEntity.class))).thenAnswer(invocation -> {
        TaskEntity entity = invocation.getArgument(0);
        entity.setId(100L);
        return entity;
    });

    var result = service.createTask(new CreateTaskRequest(
            "Не работает почта",
            "Outlook не принимает входящие письма",
            null,
            null,
            null,
            LocalDateTime.now().plusDays(1)
    ));

    assertEquals(Status.NEW, result.getStatus());
    assertEquals(Priority.MEDIUM, result.getPriority());
    assertEquals(Category.GENERAL, result.getCategory());
    verify(slaService).ensureForTicket(any(TaskEntity.class));
    verify(embeddingService).upsertTaskEmbedding(any(TaskEntity.class));
    verify(notificationService).notify(any(User.class), eq("Обращение создано"), anyString());
}
```

Запуск модульных тестов выполняется командой:

```bash
./mvnw test
```

Для получения отчёта в IDE используется стандартный JUnit runner. При необходимости отчёт можно дополнить Maven Surefire XML-файлами из каталога `target/surefire-reports` или подключить JaCoCo для расчёта покрытия строк и ветвей.

## 2.6.2 Интеграционное тестирование REST API (Postman/Newman)

Интеграционные проверки REST API выполнялись через Postman-коллекции. Коллекция группируется по функциональным областям системы:

1. **Аутентификация**:
   - `POST /auth/login` — успешный вход тестовых пользователей `user1`, `operator1`, `admin1`;
   - `POST /auth/refresh` — обновление access-токена;
   - `POST /auth/logout` — отзыв refresh-токена;
   - негативные сценарии: неверный пароль, отсутствующий refresh-токен, повторное использование отозванного токена.
2. **Заявки**:
   - создание обращения пользователем;
   - просмотр списка и карточки обращения;
   - назначение исполнителя оператором;
   - изменение статуса, закрытие, эскалация;
   - негативные сценарии: попытка закрытия без комментария, недоступный переход статуса, обращение к чужой заявке пользователем.
3. **AI API**:
   - `POST /ai/classify` — классификация текста обращения;
   - `POST /ai/similar` — поиск похожих обращений;
   - `POST /ai/recommend` — генерация рекомендации и шагов решения;
   - негативные сценарии: пустой текст, некорректные входные данные, недоступность внешнего LLM при включённом fallback.
4. **База знаний, SLA и администрирование**:
   - получение статей базы знаний;
   - просмотр SLA-отчётов;
   - просмотр аудита и справочников администратором;
   - проверка ограничений доступа для ролей `USER`, `OPERATOR`, `ADMIN`.

Postman-проверки покрывают как позитивные, так и негативные сценарии. Для автоматизированного запуска коллекции может использоваться Newman:

```bash
newman run support-system.postman_collection.json \
  --environment local.postman_environment.json \
  --reporters cli,html \
  --reporter-html-export target/newman/support-system-report.html
```

В отчёте Newman фиксируются количество выполненных запросов, успешные/ошибочные assertions, время ответа и итоговый статус прогона. HTML-отчёт удобно прикладывать к материалам ВКР как подтверждение интеграционного тестирования REST API.

## 2.6.3 Тестирование интеллектуального модуля

Интеллектуальный модуль тестировался отдельно, поскольку его качество зависит не только от корректности Java-кода, но и от устойчивости обработки ответов LLM, качества классификации, поиска похожих обращений и RAG-пайплайна.

Проверялись следующие сценарии:

- **Классификация обращений**:
  - текст про проблемы доступа должен относиться к категории `ACCESS`;
  - критичные инциденты должны получать повышенный приоритет `HIGH` или `URGENT`;
  - ответ модели с несколькими JSON-фрагментами должен приводиться к последнему валидному объекту;
  - фигурные скобки внутри JSON-строк не должны ломать парсер.
- **Поиск похожих обращений**:
  - запросы с синонимами, например `email не работает` и `Проблема с электронной почтой Outlook`, должны находить один и тот же исторический инцидент;
  - при недоступности `pgvector` используется локальный fallback-поиск по сохранённым embedding-записям;
  - результат ограничивается параметром `topK` и сортируется по убыванию score.
- **RAG-пайплайн и рекомендации**:
  - рекомендация формируется на основе статей базы знаний и похожих закрытых обращений;
  - в ответ включается объяснимость: режим работы, источники, статус LLM и исходный вывод модели;
  - при сетевой ошибке, превышении квоты или отсутствии ключа внешнего LLM пользователь получает безопасную fallback-рекомендацию вместо необработанного исключения.
- **Защита от вредоносных и некорректных строк**:
  - вход с prompt-injection фразами не должен менять формат ответа классификатора;
  - пустой или слишком короткий текст отклоняется на уровне валидации REST API;
  - некорректный JSON от модели не должен приводить к аварийному завершению сценария, вместо этого используется fallback-результат или ручная обработка оператором.

Для интеллектуального модуля применяются как автоматические JUnit-тесты с подменой `LlmClient`, так и ручные интеграционные сценарии через Postman. Такой набор проверок подтверждает, что система корректно работает при штатном ответе модели и сохраняет управляемое поведение при сбоях внешнего AI-сервиса.
